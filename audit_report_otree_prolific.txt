Developer Report: oTree 5.x + Prolific Repository Audit
Project: experimentalApp_New
Date: 2026-03-11

Scope
This report summarizes structural logic issues that can explain:
- participants returning after partial completion
- blocked cohort slots preventing replacements
- partial datasets in export
- timing values recorded as zero
- inconsistent state after Prolific return/time-out

Summary of Most Likely Causes

1. Schedule rows are deleted on participant return/time-out
Files:
- prolific_webhook.py
- img_desc/__init__.py
Functions:
- prolific_webhook_view()
- ProlificStatusGate.get()
- delete_schedule_for_participant()
- custom_export()

Why this is risky:
- The export reads experimental data from ScheduleItem rows.
- Both the webhook and the in-app Prolific gate delete ScheduleItem rows for bad Prolific statuses.
- This means returned participants can lose the very rows that store production and interpretation data.

Likely failure scenario:
- Participant completes part of the experiment.
- Some ScheduleItem rows already contain production data.
- Prolific marks the submission RETURNED or TIMED-OUT.
- The webhook or ProlificStatusGate calls delete_schedule_for_participant().
- Export later shows missing interpretation rows or missing schedule rows entirely.

Suggested fix:
- Do not delete ScheduleItem rows on return/time-out/rejection.
- Keep all collected data immutable once written.
- Add a participant/session status flag such as returned/timed_out/rejected and exclude those cases analytically in export if needed.

2. Prolific ID can be overwritten or lost, causing participant_not_found
Files:
- start/__init__.py
- img_desc/__init__.py
- prolific_webhook.py
Functions:
- _ProlificCaptureMixin._capture()
- _store_prolific_on_participant()
- CaptureProlificID.vars_for_template()
- _find_participant_by_prolific_id()

Why this is risky:
- start app captures Prolific parameters robustly.
- img_desc.CaptureProlificID writes participant.vars["prolific_id"] directly from request GET parameters and only checks uppercase names.
- If the later request does not contain those parameters, or contains lowercase keys, an existing Prolific ID can be overwritten with None/empty.
- Then the webhook cannot match the participant and does not free the slot.

Likely failure scenario:
- Participant enters via start and gets a valid prolific_id.
- On entering img_desc, the request lacks PROLIFIC_PID or uses lowercase names.
- CaptureProlificID overwrites prolific_id with an empty value.
- A return webhook arrives later and _find_participant_by_prolific_id() fails.
- Log shows "Webhook: participant_not_found".
- The cohort slot remains active and blocks replacements.

Suggested fix:
- Remove img_desc.CaptureProlificID, or make it write only when the incoming value is non-empty.
- Accept both uppercase and lowercase query keys consistently.
- Consider storing participant.label and participant.vars["prolific_id"] only once at entry.

3. Cohort slots can leak when return/time-out handling misses the participant
Files:
- prolific_webhook.py
- img_desc/__init__.py
Functions:
- prolific_webhook_view()
- _find_participant_by_prolific_id()
- free_slot_for_participant()
- assign_slot_if_needed()
- _cohort_has_free_slot()
- cohort_complete()

Why this is risky:
- Slot freeing depends on successful participant lookup from the webhook.
- If lookup fails, active CohortSlot rows are never cleared.
- assign_slot_if_needed() treats active rows as occupied.
- A cohort that is full but incomplete causes newcomers to wait indefinitely.

Likely failure scenario:
- Returned participant still has an active slot row.
- free_slot_for_participant() is never called because webhook lookup fails.
- assign_slot_if_needed() sees no free slot in that cohort.
- Since cohort_complete() is false, new participants are forced into wait state instead of replacing the missing participant.

Suggested fix:
- Make webhook participant resolution more robust.
- Store a stable mapping from Prolific participant/submission IDs to oTree participant_code.
- Add admin/audit logging for stale active slots.
- Add a recovery job or manual admin command that frees slots for returned/timed-out participants even if webhook lookup failed earlier.

4. Production data can be overwritten by replacements
Files:
- img_desc/__init__.py
Functions:
- set_sentence()
- sentence_key()
- sentences_for_line_from_trialrow()
- Q.before_next_page()
- _resolve_lookup()

Why this is risky:
- SentenceStore keys are based only on:
  exp_num | producer_slot | interpreter_slot | condition
- They do not include participant_code, round_number, trial identifier, or a stable schedule row ID.
- If a replacement participant is assigned the same slot, later production can overwrite earlier production for the same logical position.

Likely failure scenario:
- Original participant produces a sentence for a slot and then returns.
- Replacement enters the same cohort slot.
- Replacement later writes a new sentence to the same SentenceStore key.
- Interpretation/export may read the replacement sentence instead of the original participant's sentence.

Suggested fix:
- Key stored production by a unique trial identity, not only by slot.
- Include participant_code and round_number, or a TrialRow primary key / schedule row ID.
- Never allow replacement participants to overwrite previously collected participant responses.

5. Slot assignment and schedule construction are race-prone
Files:
- img_desc/__init__.py
Functions:
- assign_slot_if_needed()
- ensure_schedule_built()
- build_schedule_for_participant()
- get_schedule_item()

Why this is risky:
- assign_slot_if_needed() does a read-then-create flow without transactional protection.
- ensure_schedule_built() only checks whether round 1 exists, then builds the full schedule.
- There is no uniqueness constraint on ScheduleItem(participant_code, round_number) or CohortSlot(exp_num, slot, active).
- Concurrent requests can create duplicate assignments or duplicate schedule rows.

Likely failure scenario:
- Two requests for the same participant arrive close together.
- Both see no schedule row for round 1.
- Both build schedules and insert duplicate ScheduleItem rows.
- Reads later use qs[0], so the chosen row is arbitrary.

Suggested fix:
- Make slot allocation atomic with database constraints or transactional get-or-create semantics.
- Add uniqueness guarantees for:
  - active cohort slot identity
  - schedule row identity per participant and round
- Add a persistent flag such as schedule_built_at or schedule_version and never rebuild once response collection has started.

6. Timing variables can legitimately become zero when client_start_ts is missing
Files:
- img_desc/templates/img_desc/Q.html
- img_desc/__init__.py
Functions:
- Q.before_next_page()

Why this is risky:
- client_start_ts is injected in client-side JavaScript on page mount.
- If JavaScript fails, the page is restored from browser state, or the hidden field is absent on submission, before_next_page() sets start_decision_time = end_decision_time.
- decision_seconds then becomes 0.

Likely failure scenario:
- Participant opens the page but the hidden timer field is not populated.
- Form submits successfully.
- Server falls back to identical start/end timestamps.
- Export records zero seconds.

Suggested fix:
- Record server-side page-enter timestamp when the page is rendered.
- Use client_start_ts only as an optional override or cross-check.
- Log missing timing payloads for diagnostics.

7. Bad-status matching may miss some webhook variants
Files:
- img_desc/__init__.py
- prolific_webhook.py
Functions:
- prolific_webhook_view()

Why this is risky:
- BAD_PROLIFIC_STATUSES is matched exactly against {"TIMED-OUT", "RETURNED", "REJECTED"}.
- If Prolific sends a variant such as TIMED_OUT or different casing, the bad-status path is skipped.

Suggested fix:
- Normalize incoming status values before comparison.
- Example: uppercase + replace "_" with "-".

High-Priority Recommendations

1. Remove all schedule deletion on participant return/time-out/rejection.
2. Make Prolific identity capture write-once or non-destructive.
3. Make webhook participant lookup robust and auditable.
4. Redesign SentenceStore keys so replacements cannot overwrite earlier data.
5. Add uniqueness/transaction protection around CohortSlot and ScheduleItem creation.
6. Persist a schedule-built marker and never rebuild after data collection begins.
7. Use server-side timing as the fallback baseline instead of forcing zero durations.

Bottom Line
The strongest root cause is that return/time-out handling currently mixes two incompatible goals:
- freeing a cohort slot for replacement
- deleting/resetting participant runtime state

Freeing the slot is correct.
Deleting schedule-backed data is not.

That combination can explain partial exports, lost interpretation data, stale blocked slots, and stalled cohorts.
