**Follow-Up Report**

The updated `img_desc/__init__.py` is still not safe for production. The main failure modes from the previous audit remain open, so the repository can still lose schedule-backed data, leak cohort slots, overwrite stored production, and record zero timings.

The must-fix files are img_desc/__init__.py and prolific_webhook.py.

**Critical findings**

1. `ScheduleItem` rows are still deleted for returned/timed-out participants.
Files:
- [img_desc/__init__.py](C:\Users\benz\OneDrive - Leibniz-ZAS\Projekte\AIBR_Experiments\experimentalApp_New\img_desc\__init__.py#L280)

Functions:
- `delete_schedule_for_participant()`
- `ProlificStatusGate.get()`
- `prolific_webhook_view()`
- `custom_export()`

Failure scenario:
- A participant completes some rounds.
- Prolific marks them `RETURNED` or `TIMED-OUT`.
- The app frees the slot and deletes `ScheduleItem` rows.

Required fix:

- `img_desc` later receives a request without `PROLIFIC_PID`, or with lowercase params.
- `CaptureProlificID` writes `None`/empty into `participant.vars["prolific_id"]`.

Required fix:

**Recommended release blocker list**
- Remove all schedule deletion on participant return/time-out.
- Make Prolific ID capture non-destructive or remove the duplicate capture page.
- Redesign sentence storage keys to prevent overwrite by replacements.
- Make slot assignment and schedule creation idempotent and atomic.

**Bottom line**

The current update does not close the original production risks. The app should not be considered safe for a live Prolific run until the destructive schedule deletion and Prolific ID overwrite paths are removed at minimum.
