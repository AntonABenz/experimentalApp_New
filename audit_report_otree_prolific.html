Developer Report: oTree 5.x + Prolific Repository Audit

Project: experimentalApp_New
Date: 2026-03-11

Scope

This report summarizes structural logic issues that can explain partial completion, blocked cohort slots, missing interpretation data, zero timing values, and inconsistent state after participant return/time-out.

1. Schedule Rows Are Deleted On Return/Time-out

Files: prolific_webhook.py, img_desc/__init__.py

Functions: prolific_webhook_view(), ProlificStatusGate.get(), delete_schedule_for_participant(), custom_export()

Failure scenario: participant produces data, then is marked RETURNED/TIMED-OUT, and ScheduleItem rows are deleted even though export reads from them.

Suggested fix: never delete ScheduleItem rows on return/time-out; store status flags instead.

2. Prolific ID Can Be Overwritten Or Lost

Files: start/__init__.py, img_desc/__init__.py, prolific_webhook.py

Functions: _ProlificCaptureMixin._capture(), _store_prolific_on_participant(), CaptureProlificID.vars_for_template(), _find_participant_by_prolific_id()

Failure scenario: a valid prolific_id captured in start is later overwritten with an empty value in img_desc, causing webhook lookup failure and stale slots.

Suggested fix: remove the second capture step or make it non-destructive and case-insensitive.

3. Cohort Slot Leakage Can Block Replacements

Files: prolific_webhook.py, img_desc/__init__.py

Functions: prolific_webhook_view(), _find_participant_by_prolific_id(), free_slot_for_participant(), assign_slot_if_needed(), cohort_complete()

Failure scenario: webhook cannot resolve participant, so active CohortSlot rows are never freed and new entrants wait forever in a full-but-incomplete cohort.

Suggested fix: strengthen participant mapping, add stale-slot diagnostics, and support slot recovery.

4. Replacement Participants Can Overwrite Earlier Production

Files: img_desc/__init__.py

Functions: set_sentence(), sentence_key(), sentences_for_line_from_trialrow(), Q.before_next_page(), _resolve_lookup()

Failure scenario: SentenceStore keys are based only on experiment/slot/condition, so a replacement in the same slot can overwrite previously stored sentences.

Suggested fix: key production data by a unique trial identity such as participant + round or a stable trial row ID.

5. Slot Assignment And Schedule Construction Are Race-Prone

Files: img_desc/__init__.py

Functions: assign_slot_if_needed(), ensure_schedule_built(), build_schedule_for_participant(), get_schedule_item()

Failure scenario: concurrent requests can create duplicate slot assignments or duplicate schedule rows because creation is not atomic and there are no uniqueness guarantees.

Suggested fix: add transactional allocation and unique constraints for active slots and participant-round schedule rows.

6. Timing Values Can Become Zero

Files: img_desc/templates/img_desc/Q.html, img_desc/__init__.py

Functions: Q.before_next_page()

Failure scenario: if client_start_ts is not populated by JavaScript, the code sets start and end time to the same value and records decision_seconds = 0.

Suggested fix: record server-side page-enter timestamps and treat client timing as optional enrichment.

7. Bad-status Matching Is Too Fragile

Files: img_desc/__init__.py, prolific_webhook.py

Functions: prolific_webhook_view()

Failure scenario: exact matching against TIMED-OUT, RETURNED, and REJECTED may miss format variants.

Suggested fix: normalize incoming status strings before comparison.

Recommended Fix Order

  • Stop deleting ScheduleItem rows on return/time-out/rejection.
  • Prevent destructive overwrites of participant.vars["prolific_id"].
  • Normalize webhook statuses and harden webhook-to-participant lookup.
  • Redesign sentence storage keys to prevent overwrite by replacements.
  • Make slot allocation and schedule creation atomic and idempotent.
  • Add server-side timing fallback.

Bottom Line

The main structural problem is that return/time-out handling currently frees slots by also deleting schedule-backed data and resetting state. Slot release is correct; deletion of collected experiment data is not. That combination is the strongest explanation for partial exports, missing interpretation data, stale occupied slots, and stalled cohorts.
