Project: experimentalApp_New

Purpose
These changes are optional.
They do not fully serialize slot allocation across processes, but they are simple and reduce accidental duplicate schedule creation and duplicate active-slot assignment.

Use these only if you want a small safety improvement without a larger refactor.

def get_schedule_item(obj, participant_code: str, round_number: int):
    root = _root_subsession(obj)
    if not root or not participant_code or not round_number:
        return None

    # Sort so duplicate rows, if they ever exist, resolve consistently.
    qs = ScheduleItem.filter(
        subsession=root,
        participant_code=participant_code,
        round_number=int(round_number),
    )
    qs.sort(key=lambda it: int(getattr(it, "id", 0) or 0))
    return qs[0] if qs else None
```

1D. Add lightweight duplicate logging for easier diagnosis

Add this helper near the schedule utilities:

```python
def log_duplicate_schedule_rows(obj, participant_code: str):
    root = _root_subsession(obj)
    if not root or not participant_code:
        return

    # Count duplicates by round so logs reveal accidental double-builds.
    counts = {}
    rows = ScheduleItem.filter(subsession=root, participant_code=participant_code)
    for row in rows:
        rn = int(getattr(row, "round_number", 0) or 0)
        counts[rn] = counts.get(rn, 0) + 1

    duplicates = [rn for rn, count in counts.items() if count > 1]
    if duplicates:
        logger.warning(
            "Duplicate ScheduleItem rows detected for participant=%s rounds=%s",
            participant_code,
            duplicates,
        )

======================================================================
2. What this does and does not fix
======================================================================

These changes help with:
- duplicate `ScheduleItem` rows for the same participant/round
- duplicate active-slot rows from very close requests
- nondeterministic reads when duplicates already exist

These changes do not fully fix:
- true cross-process race conditions on multiple web workers
- the need for database-level uniqueness or transactional slot allocation

Given your setup, this note is optional.
If participant completions are rarely simultaneous, the earlier paste-fix file is much more important than this one.
